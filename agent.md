## 1. GitHub / 项目路径稳定性修复

### 发现的问题

后端原来有些路径依赖“当前运行命令时所在的目录”。如果从不同目录启动项目，或者以后从 GitHub 重新克隆到别的位置，可能出现：

- 找不到 `.env` 配置文件；
- 找不到 `frontend` 前端目录；
- 找不到 `covers` 封面目录；
- 找不到 `avatars` 头像目录；
- 找不到 `global_regions.json` 地区数据文件。

通俗理解：程序原来是“看你站在哪里找东西”，而不是“从项目自己的位置找东西”。

### 修改内容

涉及文件：

- `backend/main.py`
- `backend/scanner.py`
- `rebuild_covers.py`

主要改动：

- 增加稳定的项目路径基准，例如后端目录、项目根目录、前端目录、封面目录、头像目录；
- `.env` 改为从后端目录固定读取；
- 静态资源目录改为绝对路径挂载；
- `scanner.py` 不再写死 `D:\MusicCloud\backend\covers`；
- 修复 `rebuild_covers.py` 文档字符串中的 Windows 路径转义警告。

### 效果

项目从不同目录启动时更稳定，将来换文件夹或重新克隆后也不容易因为路径问题崩溃。

## 2. 歌词滚动低配电脑卡顿优化

### 发现的问题

低配电脑播放歌曲时，歌词每滚到下一句都会明显掉帧。主要原因是：

- 每次歌词变化都反复搜索整片歌词 DOM；
- 移动端滚动时频繁读取页面布局尺寸；
- 桌面端使用平滑滚动，动画容易叠加；
- 当前歌词高亮时改变字号，导致页面重新排版。

通俗理解：原来歌词每跳一句，都像是“重新整理一次整张桌子”，所以低配电脑会卡。

### 修改内容

涉及文件：

- `frontend/app.js`
- `frontend/style.css`

主要改动：

- 加载歌词时缓存每一行歌词元素；
- 切换高亮时只处理上一句和当前句；
- 使用二分查找定位当前歌词，避免从头扫到尾；
- 去掉歌词区的平滑滚动叠加；
- 移动端滚动不再频繁调用 `getBoundingClientRect()`；
- 当前歌词不再通过改变字号放大，改为使用 `transform: scale(...)`，减少页面重排。

### 效果

歌词滚动对 CPU 和页面排版的压力明显降低，低配电脑上应该更稳定。

## 3. 底部节律条 / 频谱条冲顶优化

### 发现的问题

网页底端的节律条会跟着音乐上下舞动，但歌曲音量较大时，频谱值很容易接近最大值，导致柱子直接冲到顶部。这样看起来不是“有节律地跳”，而是“全都顶满”，层次感变差。

### 修改内容

涉及文件：

- `frontend/app.js`
- `frontend/style.css`

主要改动：

- 给节律条增加自动音量适配：
  - 大音量时自动压低显示上限；
  - 小音量时自动抬起细节；
  - 给顶部保留空间，避免柱子一直冲顶。
- 频段读取从单点采样改为区间平均，显示更稳定；
- 加入轻微高频补偿，避免只有低频在动；
- 使用“小信号放大、强信号压缩”的曲线，让细节更明显；
- 改成“上升快、回落慢”的缓动方式，让节律条更有弹性；
- 避免每一帧都重设画布尺寸，减少额外性能开销；
- 提高节律条透明度，让视觉更明显。

### 效果

节律条不容易再冲顶，同时会比之前更灵活、更有层次，低音、高音和节奏变化更容易被看出来。
